# 边界缺陷检查清单

> [English](boundary-checklist.md) | **简体中文**

本文档持续收录在本仓库中发现的边界（boundary）/极端情况（edge case）缺陷的*类别*，
以避免同类问题被反复引入。每个条目都是一个可复用的模式：反模式（anti-pattern）、
修复方式，以及实际发现问题的位置。可以从两个角度使用：

- **编写**解析 / 匹配 / 光标 / 缓存 / 序列化 / 生命周期 /
  分类相关的代码时——提交前先核对相关条目。
- **评审**涉及上述领域的变更时——把这份清单当作检查表逐条过一遍。

这类缺陷大多能通过 `typecheck`，甚至能通过正常路径（happy path）的测试。它们只在
特定的边界输入上暴露，因此需要有意识地专门审视，而不能只看测试是否全绿。

---

## 1. `parse` 系列函数会返回 `NaN`，而所有与 `NaN` 的比较结果都是 `false`

`Date.parse(x)`、`parseInt(x)`、`parseFloat(x)`、`Number(x)` 在输入非法时返回 `NaN`。
`NaN >= cutoff`、`NaN < limit`、`NaN === n` **全部**为 `false`，因此写成「较新 → 保留」
的守卫逻辑在遇到无法解析的输入时会悄悄走进 *else* 分支。

- **正确做法：** `const n = Date.parse(x); if (Number.isNaN(n) || n >= cutoff) …` ——
  明确决定「无法解析的值」意味着什么，并显式处理。
- **发现位置：** `packages/core/src/memory/compact.ts` —— 损坏的 `addedAt` 使
  `NaN >= cutoff` 为 false，导致一条年龄未知的记忆事实被悄悄归档。
- **同类问题：** 会话元数据接受带时区偏移的时间戳，却按其原始字符串排序；应先把
  时间解析为 epoch 值，再让 `keepLast` 决定裁剪时保留哪些内容。

## 2. 前缀匹配必须带分隔符边界

`subject.startsWith(match)` 会让 `src/foo` 匹配到 `src/foobar.ts`，让
`npm run build` 匹配到 `npm run build-all` —— 一个共享前缀的「兄弟」就这样混过了关卡。

- **正确做法：** 对于**允许 / 授权**类判定，必须要求边界：`subject ===
  match`，或者 match 本身已经以分隔符结尾，或者 `subject[match.length]` 是分隔符
  （命令用 `" "`，路径用 `/` 或 `path.sep`）。对于**拒绝**类判定，则保留宽泛的
  `startsWith` —— 拒绝规则过度匹配属于失败关闭（fail closed），是安全的。
- **发现位置：** `packages/core/src/tools/permissions.ts`（`ruleMatches` /
  `boundaryPrefix`）；同样的道理此前已经体现在 `sessionAllowed` 中。
- **同类问题：** `packages/core/src/hooks/index.ts`（`hookApplies`）—— hook 的
  `pattern: "npm run build"` 匹配到了 `npm run build-all`，`src/foo` 匹配到了
  `src/foobar.ts`。
- **同类问题：** `apps/tui/src/app.tsx`（`/memory edit`）—— 裸的
  `target.startsWith(memoryDir)` 让 `../memory2/project.md` 通过了检查，因为
  `memory2` 与之共享同一个字符串前缀。
- **同类问题：** `packages/core/src/agent/trace.ts` —— 会话 id 被直接拼接进
  读/写/删除路径，依赖每个调用方自行在到达 Core 之前拒绝路径穿越。
- **同类问题：** `packages/core/src/agent/trace.ts` —— rewind 检查点路径使用
  词法包含判断，可以通过父目录中的符号链接逃逸。
- **同类问题：** `packages/core/src/memory/store.ts` —— `@import` 与根记忆文件
  使用词法包含判断，但读取会跟随符号链接跳出工作区。
- **同类问题：** `apps/server/src/files.ts` —— 上传目录本身可能是指向工作区之外
  的符号链接，即便返回的相对路径看起来是安全的。
- **同类问题：** `packages/core/src/skills/manage.ts` —— enable/disable/remove
  直接拼接未经校验的技能 id，可能修改技能根目录之外的目录。
- **同类问题：** `apps/cli/src/authorized-dirs.ts` —— 祖先目录匹配使用了硬编码的
  分隔符，无法正确处理 Windows 路径和文件系统根目录。

## 3. 缓存 / memo 的键必须包含所有影响输出的输入

如果两个不同的请求哈希到同一个键，第二个请求会悄悄拿到第一个请求的答案。

- **正确做法：** 对**所有**影响输出的字段做哈希，而不仅是那些显眼的字段。
- **发现位置：** `packages/core/src/provider/cache.ts` —— 缓存键遗漏了
  `temperature` 和 `maxTokens`，导致后续用更大 `maxTokens` 的调用重放了之前
  被截断的回复。
- **同类问题：** `apps/tui/src/statusline-scheduler.ts` —— 缓存键遗漏了状态栏
  命令本身，替换或重新启用该命令后仍在复用过期输出。
- **同类问题：** 仅按模型/请求做键的 provider 响应缓存可能跨端点或跨租户重放
  数据；应把每一个影响响应的 provider 设置纳入一个不透明的身份标识。
- **同类问题：** 规范化后的 MCP prompt 名称可能冲突（`foo_bar`/`foo-bar`）；
  应分配确定性的唯一命令名，并在查找时使用同一套映射。
- **同类问题：** 子代理（subagent）的派发 id 会在新的 manager/run 中从头计数。
  转写记录只能更新最新的、处于活跃状态的匹配派发，而不是恰好复用了同一个
  局部 id 的某张已完成的旧卡片。
- **同类问题：** 把源码行号编进 Finding id 会导致无关行插入后同一个漏洞被
  重复报告。应优先使用稳定的规则、路径和规范化证据构成身份，行号仅作为位置信息。

## 4. 序列化与反序列化必须严格互逆

如果写入方使用 `JSON.stringify`，而读取方只是剥掉外层引号，那么任何含 `"` 或 `\`
的值在「渲染→重新加载」的往返中都会被破坏。

- **正确做法：** 有意识地配对编码器和解码器；写入方做了 JSON 编码，读取方就必须
  `JSON.parse`。添加一个含引号/反斜杠值的往返（round-trip）测试。
- **发现位置：** `packages/core/src/subagents/frontmatter.ts` 与 `import.ts`。
- **同类问题：** Git 路径可能包含换行符，因此按行分割的 worktree porcelain 输出
  是不可逆的；应使用 `git worktree list --porcelain -z` 并端到端解析
  NUL 分隔的字段。
- **同类问题：** 结构化数字语法中的别名必须在解析出值之后再做规范化，而不是在
  源字符串上做字符替换；cron 中把 DOW 的 `7` 直接替换会破坏 `5-7`、`*/7`
  这类合法的区间和步长写法。

## 5. 光标 / 索引运算必须感知代理对与多字节字符

星光平面字符（emoji、CJK 扩展区）占两个 UTF-16 码元。裸的 `cursor ± 1` 会落在
两半*之间*，下一次编辑时就会破坏文本。

- **正确做法：** 按完整码点（code point）步进（`stepLeft`/`stepRight`/`moveLeft`/`moveRight`），
  对任何被钳制过的位置执行 `snapToBoundary`。用 `"😀"` 做测试。
- **发现位置：** `apps/tui/src/vim.ts`（插入模式 Escape、字符级 `p`）——
  这些辅助函数在 `editor.ts` 里早已存在，只是 vim 绕开了它们。
- **同类问题：** `apps/tui/src/components/MultilineComposer.tsx` —— 光标渲染按
  单个 UTF-16 码元索引，把 emoji 代理对切成了两半。
- **同类问题：** Vim 的 word/end 移动直接用字符串索引，可能停在代理对内部；
  应使用编辑器的码点辅助函数来分类和前进。
- **同类问题：** TUI 垂直移动把 UTF-16 偏移量当作终端列号使用，标签页标题
  按字素中间截断；应计算显示宽度，且只在字素（grapheme）边界处截断。

## 6. 每个 `addEventListener` 都需要一个配对的 `removeEventListener`

`{ once: true }` 只有在监听器*触发*时才会移除它。在正常的完成/清理路径上它永远
不会触发，于是挂在长生命周期 signal 上的监听器每执行一次操作就泄漏一个。

- **正确做法：** 给处理函数命名，并在 `finally` / settle 回调中
  `removeEventListener`（或使用 `AbortSignal.any`）。定时器、流、子进程同理。
- **发现位置：** `packages/core/src/subagents/manager.ts` —— 挂在共享父级
  `AbortSignal` 上的 abort 监听器。
- **同类问题：** `packages/core/src/agent/trace.ts` —— 删除会话前必须关闭缓存的
  追加写文件描述符，否则重建同一个 id 后仍会继续写入已被 unlink 的 inode。

## 7. 在序列化 / 请求边界处强制执行协议不变量

在操作中途持久化的状态（两次写入之间发生取消 / 错误 / 触达上限）可能违反下游
消费者所要求的不变量。只修其中一条写入路径，其他所有路径依然暴露在外。

- **正确做法：** 在数据离开系统的地方集中强制执行不变量，这样无论数据是怎么
  产生的，不变量都成立。对格式良好的输入，这一步应当是空操作（no-op）。
- **发现位置：** `packages/core/src/provider/mapping.ts`（`toWireMessages`）——
  一条没有对应 `tool` 结果的 assistant `tool_calls`（回合在执行途中被取消或
  触达上限）会让 OpenAI 兼容 API 在 `/resume` 时返回 400。现在未应答的
  tool_calls 和孤立的 tool 结果会在构建请求前被丢弃。
- **同类问题：** 流式 EOF 在没有收到必需的 `[DONE]` 终止符时就被视为完成，
  把断开的连接变成了一次「成功」的部分响应。
- **同类问题：** tool-call id 在整个会话范围内不保证唯一。应在每个 assistant
  回合内部配对结果；全局的「已应答 id」集合会在早先回合复用了某个 id 时，
  让一次被中断的调用看起来像已完成。
- **同类问题：** 会话审计导出通过一个全局 id 映射来配对工具结果，导致后续回合
  复用某个 id 时改写了先前调用的证据。审计/报告代码必须保持同样的
  「按 assistant 回合配对」边界。
- **同类问题：** 一个可选的安全字段如果存在但格式非法，必须拒绝该请求。
  此前非法的 WebSocket `selectedHunks` 会把一个部分权限响应放大为对整个
  补丁的批准。
- **同类问题：** 未知的子代理模式此前默认落到可写的 `edit`；枚举型安全设置
  必须拒绝未知值，而不是选择一个宽松的兜底值。

## 8. `JSON.parse` 成功不代表你拿到的是对象

`null`、`42`、`"x"`、`[]` 都是合法的 JSON。随后展开 `...parsed.field` 的代码会
抛出一个难以定位的 `TypeError`。

- **正确做法：** 解析配置类输入后，先断言它是一个非 null、非数组的对象再使用；
  否则返回 `{}` 或抛出描述性错误。
- **发现位置：** `apps/cli/src/config.ts`（`readJson` / `readSettingsFile`）。
- **同类问题：** `apps/cli/src/mcp-config.ts`（`readConfigDoc`）—— JSON 的
  `null` / `[]` / `"x"` 被当作配置文档返回，随后在访问 `doc.mcpServers` 时崩溃。
- **同类问题：** `apps/server/src/routes/settings.ts`（`readConfigDoc` /
  `mutateMcpServers`）—— 非对象形态的项目配置 JSON 会让 `/api/hooks`、
  `/api/mcp` 等设置路由崩溃。
- **同类问题：** `apps/cli/src/commands/run.ts`（`--mcp-config`）、
  `apps/cli/src/commands/config.ts`（`config set`）以及
  `apps/server/src/config.ts`（`setConfigValue`）—— 非对象 JSON 通过了解析步骤，
  却在稍后的 MCP 合并或配置变更中失败。
- **同类问题：** `packages/core/src/runtime/client.ts` —— `null` 之类的合法 JSON
  或伪造的响应形态可能让 readline 回调崩溃，或用非法数据 settle 一个挂起的
  runtime 请求。
- **同类问题：** `packages/core/src/agent/trace.ts` —— 未经校验的会话元数据可能
  让列表功能崩溃，或伪造出一个稍后被裁剪逻辑使用的 id。
- **同类问题：** `packages/core/src/agent/trace.ts` —— `messages.jsonl` 中合法的
  JSON 标量和数组被当作伪造的 `ChatMessage` 值重放。
- **同类问题：** `packages/core/src/mcp/http.ts` —— 纯 JSON 传输接受了 `null`，
  以及属于另一个 JSON-RPC 请求 id 的响应。
- **同类问题：** `packages/core/src/provider/sse.ts` 与 `mapping.ts` —— 合法的
  JSON 非对象值会让流式处理崩溃，而非有限的 token 计数会污染成本与预算核算。
- **同类问题：** `packages/core/src/tools/lsp/client.ts` —— 帧化的 JSON `null`
  到达了 stdout 事件分发器，可能在请求 Promise 之外抛出异常。
- **同类问题：** `packages/core/src/mcp/tools.ts` —— 畸形的 `tools/list` 数据在
  后续的映射循环中逃出了按服务器隔离的失败边界。
- **同类问题：** `apps/server/src/config.ts` 与 CLI doctor —— JSON `null` 尽管
  解析本身成功，仍然进入了对象展开/属性访问。
- **同类问题：** `packages/eval-harness/src/config.ts` 把任意 JSON 强转为
  `EvalConfig`；标量会让 provider 选择崩溃，畸形的嵌套定价数据可能污染成本核算。
  应过滤标量字段并校验每一个价格。
- **同类问题：** `packages/core/src/skills/manage.ts` —— 非对象形态的 `skill.json`
  值会让 enable/disable 崩溃，而不是被修复。
- **同类问题：** `apps/tui/src/config.ts` —— 与 CLI 和 server 不同，合法的 JSON
  标量或数组被直接传入分层合并逻辑，`null` 在第一次属性访问时就让 TUI 启动崩溃。
- **同类问题：** `packages/shared/src/config-layers.ts` —— 一个形态是对象的配置
  仍可能提供非数组的权限规则、非对象的 MCP 映射，或畸形的 MCP 条目/hook，
  导致合并或下游消费者崩溃。应校验每一个结构化字段，并保留低优先级层中的有效值。
- **同类问题：** 只是安全地忽略非对象配置还不够——如果 doctor 仅仅因为
  `JSON.parse` 成功就把它报告为有效，问题依旧。配置诊断也必须校验预期的
  顶层形态。

## 9. 「只读 vs 有副作用」分类：核实每个命令的真实效果

无参数 ≠ 列表操作。裸的 `git stash` 等价于 `git stash push`，会修改工作树；
把「无参数 = 只读」的假设写进代码后，它被免确认自动执行了。

- **正确做法：** 按命令的实际副作用逐个子命令分类。拿不准时按有副作用处理
  （要求确认）—— 失败关闭（fail closed）。
- **发现位置：** `packages/core/src/tools/run-command.ts`（`classifyGit`）。
- **同类问题：** `packages/core/src/tools/run-command.ts`（`classifyGh`）——
  `gh api --method=POST` / `-XPOST` 以及 `--field=...` 形式没有被解析为
  有副作用，可能被误判为只读的 GET 请求。
- **同类问题：** 重复出现的 `gh api -X/--method` 标志——只检查第一个会让
  后面的 POST 覆盖一个已被自动批准的 GET 分类。
- **同类问题：** `apps/server/src/routes/git.ts` —— 客户端文件名被直接作为 Git
  pathspec 传入，以 pathspec 魔法字符开头的文件名会改变命令的作用范围。
- **同类问题：** 命中允许列表的前缀并不等于授权一个带未加引号控制操作符的
  shell 程序。在内置、用户、会话或规则级自动批准之前，必须先拒绝复合语法和重定向。

## 10. 钳制参与排序 / 尺寸 / 预算计算的外部数字

来自用户编写文件的无界数值可能压倒一个本该只作平局裁决（tie-breaker）的分数，
或者冲破预算。

- **正确做法：** 在加载边界处钳制到预期区间（`Math.max(lo,
  Math.min(hi, x))`），而不是信任输入。
- **发现位置：** `packages/core/src/skills/load.ts` —— 精心构造的 `priority: 500`
  压过了真实的匹配信号（priority 的设计区间是 `[0,100]`）。

## 11. 解析器要处理空 / 未初始（unborn）/ 零值状态

全新仓库（无提交）、空集合、空字符串、单元素、零次试验——这些都会产生
正常路径从未见过的输出形态。

- **正确做法：** 为每个解析器或统计函数枚举零/一/未初始的情形，并加以测试。
- **发现位置：** `apps/server/src/rest.ts`（`gitStatus`）—— `## No commits yet on main`
  把分支解析成了 `"No"`。（另见 `packages/eval-harness` 各处的空集合防护。）
- **同类问题：** 按行分割的 Git porcelain 解析把普通文件名中的 ` -> ` 当成了
  重命名语法；应使用 NUL 分隔的机器格式。

## 12. 格式化数字的符号要在舍入*之后*决定

`value >= 0 ? "+"+fixed : fixed` 对一个没有变化的差值打印 `+0`，对一个舍入到零的
微小负值打印 `-0.0000` —— 两者都有误导性。

- **正确做法：** 先舍入，如果舍入后的值是 `0`，输出无符号的零。
- **发现位置：** `packages/eval-harness/src/{report,ab}.ts`（`signed`）。

## 13. 注释/文档与测试矛盾时，以测试为准

一条文档注释声称「单方运行的任务 = 平局」，而测试断言它记在实际运行它的
variant 名下。不要为了迎合注释而「修」代码——先确认意图（测试编码了意图），
然后修注释。

- **发现位置：** `packages/eval-harness/src/ab.ts`（`AbSummary` 的文档注释 vs
  `compareVariants` 的实际行为）。

## 14. 解析出的数字元数据必须是有限值，而不仅仅是 number 类型

JSON 可以把 `1e999` 之类的巨大数字字面量解析成 `Infinity`；`typeof` 仍然是
`"number"`，但新鲜度检查和光标/索引运算都会因此出错。

- **正确做法：** 对缓存时间戳、TTL、索引、光标及其他解析出的数字元数据，
  在参与运算前用 `Number.isFinite` 校验。
- **发现位置：** `packages/core/src/provider/cache.ts` —— 非有限的缓存 `ts`
  可能让一条被投毒的缓存条目看起来始终新鲜。
- **同类问题：** `apps/cli/src/version-check.ts` —— 无穷大的 `checkedAt` 时间戳
  让更新检查缓存永远新鲜。
- **同类问题：** `apps/server/src/recents.ts` —— 无穷大的 `lastOpened` 时间戳
  永久霸占最近工作区排序的首位。
- **同类问题：** `apps/cli/src/schedule.ts` —— 持久化的任务预算为无穷大时，
  成本熔断彻底失效，因为没有任何有限花费能达到 `Infinity`。
- **同类问题：** `apps/server/src/ws.ts` —— `selectedHunks` 只校验了外层数组，
  允许负数、非整数和无界索引进入 Core。

## 15. 共享的安全守卫可能需要更窄的、按能力定制的例外

直接复用一个严格的守卫可能悄悄让新能力完全不可用。网页抓取器必须拒绝
loopback 地址，而浏览器验证工具必须能够检查一个经用户确认的本地开发服务器。

- **正确做法：** 保持严格的共享默认值，在能力边界处添加尽可能小的显式例外；
  不要放宽共享守卫本身。
- **发现位置：** `packages/core/src/tools/builtins/browser.ts` —— 复用
  `checkFetchUrl` 使文档中描述的 `http://localhost:5173` 工作流完全无法使用。

## 16. 校验每一跳网络请求，而不仅是初始 URL

一个已批准的公网 URL 可能重定向到私有地址，一个已加载的页面可能请求私有的
子资源。只检查第一个 URL 会让 SSRF 守卫在导航开始后形同虚设。

- **正确做法：** 在浏览器/网络请求边界处对重定向和子资源同样执行 URL 策略，
  而不仅是初始导航。
- **发现位置：** `packages/core/src/tools/builtins/browser.ts` —— 初始 URL 检查了，
  但 Playwright 后续跟随的请求没有重新应用该策略。
- **同类问题：** `web_fetch` 把重定向交给 `fetch(..., redirect: "follow")`，且不检查
  DNS 结果；一个看似公网的 URL 可能在单次词法检查后解析或跳转到环回地址。应拒绝
  每一个非公网解析结果，并在跟随重定向前逐跳手动校验。

## 17. 在最细粒度的可观测边界上执行护栏

只在高层迭代之间检查预算或取消信号，会让单次迭代在停止条件已经可观测之后
继续发起调用。同样，取消父操作却不向其子进程发信号，会让用户干等到超时。

- **正确做法：** 在每次用量更新时检查预算，把取消传播到活跃的子进程树中，
  并在做预检工作之前先检查 signal 是否已经处于 aborted 状态。
- **发现位置：** `packages/core/src/agent/auto-loop.ts` 和
  `packages/core/src/tools/run-command.ts` —— 循环的预算/取消此前只在一次
  完整的 agent 运行或验证超时之后才生效。
- **同类问题：** `packages/core/src/provider/http.ts` —— 调用方的取消被当作
  可重试的网络错误处理，且重试退避（backoff）忽略了 abort signal。
- **同类问题：** `packages/core/src/agent/loop.ts` —— finalize 阶段的自动验证与
  自动 lint 命令漏传了运行 signal，把取消延迟到了命令退出。
- **同类问题：** `apps/tui/src/app.tsx` —— Ctrl+C 中止了运行控制器，却让前端的
  `ask_user` Promise 悬而未决，运行本身始终观测不到取消。
- **同类问题：** agent 循环没有把它的 signal 放进 `ToolContext`，导致一个执行中
  的前台命令在取消后仍然存活，尽管执行器本身支持取消。
- **同类问题：** 一次成功的非流式 provider 响应在收到响应头后就清除了超时，
  留下一个停滞的 JSON body 无法取消；应在 body 消费完成之前保留超时和
  调用方 signal。
- **同类问题：** 响应体的消费发生在重试辅助函数返回之后，导致停滞、截断或
  畸形的 `200` body 既不重试也到不了模型回退；解析必须留在单次尝试的边界之内。
- **同类问题：** 回退（fallback）错误处理在调用方已取消回退尝试的情况下，仍然
  重新抛出主模型的失败；调用方的 abort 必须优先于保留更早的可重试错误。
- **同类问题：** `DOMException.code` 是数字，直接转发任意错误码会把一个执行中
  的 `AbortError` 分类为失败的会话；被中止的运行 signal 必须强制使用稳定的
  字符串错误码 `cancelled`。
- **同类问题：** TUI 的异步 MCP prompt 在 prompt 解析完成之后才预约运行槽位，
  且 Ctrl+C 计数是全局的；应在 await 之前先预约，并把中断状态绑定到发起它的
  标签页/运行身份上。

## 18. 把内部状态排除在收敛判断的输入之外

把编排状态持久化在工作区内部，同时又把它计入工作区指纹，会让每一次迭代
看起来都「有进展」。

- **正确做法：** 把内部 trace、上传文件和循环状态文件排除在内容指纹之外，
  并独立地限制实时输出的事件数量和分块大小。
- **发现位置：** `packages/core/src/agent/auto-loop.ts` —— 持久化的循环更新与
  无上限的验证器输出块可能破坏「无进展检测」，或让客户端无限膨胀。

## 19. 数字选项解析器必须消费整个字符串

`parseInt("2x")` 和 `parseFloat("1.5usd")` 会悄悄接受一个合法前缀。除非单独检查，
它们还会放行非有限值。

- **正确做法：** 先校验完整的数字语法，再做转换，并按需要求结果是安全整数或
  有限浮点数。
- **发现位置：** `apps/cli/src/index.ts` —— 全局的正整数/浮点数选项解析器。
- **同类问题：** `apps/cli/src/schedule.ts` —— cron 片段使用 `parseInt`，接受了
  `1x`、`1-2x`、`*/2x` 之类的值。
- **同类问题：** CLI 的 `serve --port` 和 `sessions prune` 接受了垃圾后缀，
  因为 `parseInt("12junk")` 返回 `12`。
- **同类问题：** CLI 的权限 hunk 选择和 `ask_user` 选择接受了 `1abc` 之类的
  尾随垃圾字符；必须完整校验每一个 token，并对照实际提供的索引范围，
  才能批准或选中。
- **同类问题：** eval 的 `--repeat` 使用 `Number()`，即使该选项表示十进制迭代次数，
  仍会接受十六进制和指数形式。

## 20. 生命周期清理在删除共享状态前必须先证明所有权

一个过期的 worker 可能在其租约（lease）被替换后才完成，然后误删新所有者的锁，
导致同一份持久化状态被并发修改。

- **正确做法：** 用不可猜测的所有权令牌标识租约，只回收确已死亡的所有者，
  并在清理删除锁之前再次比对令牌。
- **发现位置：** `packages/core/src/agent/loop-state.ts` —— 自治循环租约。
- **同类问题：** `apps/cli/src/loop-worktree.ts` —— 清理现在同时要求保留的
  worktree 根目录和 Loop 专用的 `seekforge/loop-*` 分支前缀，因此不会误删
  其他 SeekForge 工作流的检出目录。
- **同类问题：** 并发的首次 LSP 调用复用了一个在 initialize 握手完成之前就被
  插入注册表的会话；应共享启动 Promise。
- **同类问题：** LSP 注册表只按语言做键，导致并发的工作区互相销毁对方的
  服务器；键中必须包含工作区身份。
- **同类问题：** 缓存的 LSP 文档在定义/引用请求前没有刷新；应跟踪最近的文本，
  并在磁盘编辑后发送 `didChange`。
- **同类问题：** 同一 URI 的并发诊断请求覆盖了单一等待者；应合并进行中的请求，
  或为该身份保留所有等待者。
- **同类问题：** 顶层 agent 清理会全局销毁共享的 LSP 和浏览器进程，而并发的
  运行仍在使用它们；应为每次运行保留一个令牌，只在最后一个匹配的所有者
  释放后才拆除。
- **同类问题：** LSP 服务器退出/销毁时把挂起的诊断请求 resolve 成一个空的
  成功结果；生命周期失败必须 reject 所有受影响的等待者。
- **同类问题：** 两个连接可以同时 resume 同一个持久化会话，交错写入 JSONL、
  元数据和检查点；应获取一个运行作用域的会话租约。
- **同类问题：** 服务器关闭时只关了 HTTP 监听器，没有中止活跃的触发器运行；
  应跟踪受管的运行句柄，并在 close 之前 await 它们的清理。
- **同类问题：** webhook 投递 ID 在负载校验之前就被占用，一次畸形请求就永久
  消耗掉了一个合法的重试标识符。
- **同类问题：** 前台 shell 可能退出而某个后台后代进程仍持有输出管道，导致
  读取端的 join 绕过命令超时；应在 join 管道读取器之前先清理自己拥有的进程组。
- **同类问题：** 后代进程可以调用 `setsid()` 逃出该进程组，同时仍持有
  stdout/stderr；输出排空本身也需要截止时间，绝不能无条件地 join 一个可能
  永远看不到 EOF 的读取器。
- **同类问题：** 只在 provider 边界添加取消支持是不够的，除非 agent 循环把它的
  运行 signal 传入每一个活跃的模型请求，包括流式读取和非流式 body 消费。
- **同类问题：** 过期锁的恢复本身也需要一个独占的恢复租约；否则两个恢复者
  可能竞速，第二个在验证旧所有者后可能把第一个刚获得的新锁重命名掉。
- **同类问题：** JSON-RPC 的请求顺序不等于响应顺序。串行化所有 MCP 请求会让
  一个耗时的工具调用阻塞 ping/list/取消，尽管请求 ID 本身允许独立的
  并行处理。
- **同类问题：** 取消必须在客户端移除匹配的挂起请求、定时器和监听器，发送
  `notifications/cancelled`，并且只 abort 服务器端具有相同请求 ID 的工具上下文。
- **同类问题：** 子进程 runtime 的 stdin 循环在阻塞执行单个请求时无法观测取消；
  应通过有界的 worker 池分发，让请求 ID 在输出排空期间保持活跃，并让
  取消/EOF 设置对应命令的终止标志。
- **同类问题：** 客户端侧的 JSON-RPC 超时也是一种取消；应在 reject 之前带上
  原始请求 ID 发送 `notifications/cancelled`，避免服务器继续做已被放弃的工作。
- **同类问题：** HTTP 取消通知不能 await 到完成；否则一个无响应的服务器会在
  原始的取消或超时到达调用方之前再叠加一个完整的超时时长。
- **同类问题：** LSP 请求取消必须移除挂起的请求、定时器和 abort 监听器，
  然后带上原始 ID 发送 `$/cancelRequest`。
- **同类问题：** SSE 对端可以永远流式发送一个不终止的事件；应同时限制完整
  事件的大小和未完成缓冲区的大小，并在溢出时取消读取器，让内存占用和
  传输生命周期都有界。
- **同类问题：** 在文件系统清理成功之前就把租约标记为已释放，会把一次瞬时的
  清理失败变成一个活着的孤儿。应保持本地所有权，并让 `release()` 可重试，
  直到令牌所拥有的目录确实被删除。
- **同类问题：** 服务器关闭时中止了 socket，却没有 await 游离的 WebSocket/REST
  操作；应跟踪每一个已启动的操作并排空整个集合。
- **同类问题：** 一个共享的可变重试回调把并发的 AgentCore 运行全部路由到
  最后注册的那个队列。应把重试投递绑定到发起它的异步运行上下文。
- **同类问题：** runtime 销毁时在新派生的子进程来得及消费排队的取消消息之前
  就强制关闭了它（并行负载下）。关闭要有界，但应为有序的 stdin 消息排空
  留出一个现实的宽限窗口。

## 21. 在成本或所有权变得可观测的事件点建立检查点

只在整个 agent 运行结束后才持久化，一旦进程在 provider 事件与最终报告之间
退出，会话 id 和已计费的用量就丢了。

- **正确做法：** 立即为会话身份建立检查点，并以幂等方式持久化累计用量更新；
  最终写入应重复相同的绝对总量。
- **发现位置：** `packages/core/src/agent/auto-loop.ts` —— 崩溃恢复可能打开一个
  新会话并少算 Loop 预算。

## 22. 一旦声明请求已中断，就绝不能再重放它

重连队列的生命周期可能超过创建它的 UI 操作。事后重放该请求会启动不可见的
工作，并让控件与服务器状态脱节。

- **正确做法：** 当某个连接失败时，清空绑定在该连接上的排队请求；用户在断连
  期间有意提交的请求属于下一次连接尝试。
- **发现位置：** `apps/desktop/src/lib/ws.ts` —— 一个排队的 Loop 可能在 store
  已经清除其运行状态之后才启动。
- **同类问题：** `apps/desktop/src/store.ts` —— 在 socket 运行仍活跃时重置会话，
  让迟到的事件填充了刚被清空的转写记录。
- **同类问题：** Desktop 的 backtrack 和 continue 回调解析到的是 await 之后
  「当时活跃」的标签页/工作区，而不是 await 之前捕获的身份。
- **同类问题：** 工作区作用域的视图请求可能重绘一个新选中的工作区；应按
  工作区重新挂载视图，并使聊天作用域的回调失效。
- **同类问题：** 一次延迟的图片上传从共享组件实例读到了下一个标签页的草稿；
  异步 composer 状态必须按标签页身份做键。
- **同类问题：** 绑定标签页的主页视图接受了 workspace prop，但它的最近会话/
  技能/代理请求仍然回退到全局活跃工作区；捕获到的标签页工作区必须传入
  每一个作用域请求。
- **同类问题：** 会话、代理和技能的详情请求在选中项变化后才提交；应把每个
  响应绑定到一个 generation 和选中 id 上。
- **同类问题：** Git 状态和 hooks 加载在工作区切换后才提交，随后破坏性操作或
  保存操作作用到了新激活的工作区。应捕获工作区身份，同时守卫响应提交和变更。
- **同类问题：** TUI 的草稿和运行设置跨越了标签页边界，因为编辑器状态是全局的，
  而模型/审批模式是在 await 一次 MCP 加载之后才读取的；草稿要按标签页做键，
  所有运行输入要在第一个 await 之前快照。

## 23. PID 不是持久的进程身份

操作系统会复用 PID。一个含有「活着但被复用的 PID」的过期锁可能永久保持活跃，
而一个只写了一半的锁可能被误判为已死的所有者而被抢走。

- **正确做法：** 随 PID 一起持久化进程启动身份，恢复时两者一起比对，并在
  有限的宽限期内把新出现的畸形锁视为活跃。
- **发现位置：** `packages/core/src/agent/loop-state.ts` —— 持久化的 Loop 租约。
- **同类问题：** `apps/cli/src/schedule.ts` 在调度器租约上重复了只查 PID 的检查；
  那里同样要持久化并比对进程启动身份。
- **同类问题：** 恢复锁本身也可能因所有者崩溃而被遗弃；应给畸形的恢复状态
  一个宽限期，过期后再回收。

## 24. 统计「已完成」的单元，而不是「已开始」的单元

在 `iteration.start` 时就为迭代号建检查点，意味着进程在该迭代产出结果之前
崩溃时，这个名额已经被消耗掉了。

- **正确做法：** 可恢复的会话与成本事件应立即建检查点，但单元计数器只在
  证明完成的事件点推进。
- **发现位置：** `packages/core/src/agent/auto-loop.ts` —— 被中断的 Loop 迭代。

## 25. 追加式日志只能恢复到最长有效前缀

跳过一条畸形的 JSONL 记录并接受后续记录，可能把本来并不相邻的事件拼在一起，
在一次部分写入之后破坏协议顺序。

- **正确做法：** 在第一条畸形记录处停止，只重放最长的有效前缀；元数据替换
  要用「临时文件 + rename」保证原子性。
- **发现位置：** `packages/core/src/agent/trace.ts` —— 会话恢复 trace。
- **同类问题：** 检查点恢复跳过了畸形行并接受了后面的快照，使 rewind 信任了
  因果前缀缺失的状态。

## 26. 逻辑路径相等不等于物理工作区相等

符号链接别名以及 `/var` 与 `/private/var` 这类平台别名可以指向同一个目录，
却通不过字符串相等检查。

- **正确做法：** 在持久化、给租约做键或校验加载状态之前，先用 `realpath`
  规范化已存在的工作区根目录。
- **发现位置：** `packages/core/src/agent/loop-state.ts` —— 非 Git 的 Loop 管理。

## 27. 有界的文件前缀不是内容指纹

只哈希第一个分块加文件大小，会漏掉大文件后部的等长修改，从而触发错误的
「无进展」停止。

- **正确做法：** 以有界内存的分块方式，让完整文件流过哈希。
- **发现位置：** `packages/core/src/agent/auto-loop.ts` —— Loop 收敛检测。

## 28. 不仅要校验操作数，还要校验运算结果

两个有限的正数相加/相乘可以溢出为 `Infinity`。如果后续某一层把非有限值当作
「未设置」处理，一次溢出就会悄悄拆掉一道护栏。

- **正确做法：** 对产生预算、上限、时间戳或尺寸的加法和乘法，校验其结果。
- **发现位置：** `packages/core/src/agent/auto-loop.ts` —— 累加型 resume 预算。
- **同类问题：** 调度间隔的计数是有限的，但换算成毫秒后可能超出安全整数范围；
  乘积也要校验。

## 29. 元数据调用可能跟随符号链接跨越沙箱边界

`stat` 会跟随符号链接，因此本意是给工作区条目做指纹的代码，可能意外读取了
工作区之外的目标。而完全忽略符号链接又会漏掉链接本身的变化。

- **正确做法：** 用 `lstat` 分类条目，对符号链接哈希其 `readlink` 输出；
  绝不把链接目标当作工作区内容打开。
- **发现位置：** `packages/core/src/agent/auto-loop.ts` —— 收敛指纹计算。
- **同类问题：** 缓存的服务器搜索结果和项目配置/触发器路径在重新打开时没有
  做物理路径复验，后续的符号链接替换可以逃出工作区；应拒绝符号链接并用
  `O_NOFOLLOW` 打开。
- **同类问题：** 内部状态根目录需要比普通工作区解析更严格的语义。会话 trace
  和 Git worktree 根目录必须拒绝任何含符号链接的目录组件，复验物理包含关系，
  并在读写之前使用 no-follow 的叶节点打开或原子替换。
- **同类问题：** 共享临时目录下的可预测根目录本身就是一道安全边界。创建每一级
  组件时都不跟随符号链接，并要求当前操作系统用户所有加私有（`0700`）目录权限。
- **同类问题：** 「先校验路径、稍后再重新打开」会留下替换窗口。应以 no-follow
  标志打开叶节点和父目录，比对描述符/路径身份，并把截断推迟到打开的文件
  通过物理复验之后。
- **同类问题：** 任务的 `@path` 展开在 `stat` 和 `readFile` 之前只做了词法包含
  判断，允许工作区或额外只读目录内的符号链接注入该根之外的文件。应对两个
  根目录和被引用文件都做物理解析，读取前再次检查包含关系。
- **同类问题：** 无认证的静态文件服务在其根目录内部跟随了符号链接。应规范化
  静态根目录，并对每一个被请求的资源使用 no-follow 描述符读取加
  路径/描述符身份检查。
- **同类问题：** 服务器的原始上传边界把符号链接化的上传目录的物理目标当作
  可信目录，且目录列举后重新打开一个已验证的路径时没有再检查身份。
- **同类问题：** 项目 Skill 发现打开了指向工作区之外的符号链接 `SKILL.md`；
  应对照其所属根目录校验物理内容文件。

## 30. 相关的变更必须共享同一个串行化域

对变更同一底层资源的操作使用各自独立的锁，并不能防止竞态。Git worktree 的
create 与 merge、remove 修改的是同一份基础仓库元数据，尽管它们的 API 路由和
目标 id 各不相同。

- **正确做法：** 找出被变更的物理资源，按该身份建一把锁，并让这一族的每个
  操作都获取同一个工作区/会话守卫。
- **发现位置：** `apps/server/src/worktrees.ts` —— create、merge、remove 现在共享
  基础仓库锁；create 还会持有基础工作区守卫。
- **同类问题：** REST 的 stage/unstage/discard/commit 在该锁之外变更同一份
  index 和 refs；Git 路由与 worktree 操作必须使用按物理公共 Git 目录做键的
  同一个协调器。
- **同类问题：** `PUT /api/file` 在没有活跃会话守卫的情况下写入工作区文件，
  允许编辑器覆盖 Agent 的并发修改。每一个独立的工作区变更入口都必须获取
  同一个守卫。
- **同类问题：** Desktop 的 Git stage/unstage/discard 与 commit 使用了各自独立的
  pending 标志，允许并发发起相互冲突的写操作。
- **同类问题：** backtrack 把恢复文件和截断 trace 作为两次独立变更执行，
  记忆压缩在归档持久化之前就删除了事实。应先执行可能失败的前置步骤，
  再在持有共享守卫的情况下提交破坏性更新。
- **同类问题：** TUI 的 `/rewind` 缺少相邻历史命令都有的活跃运行守卫，
  允许检查点恢复与 Agent 的写入竞速。

## 31. 语法的派生语义应基于解析后的值

文本写法不等于语义限制。`*` 和 `*/1` 覆盖的 cron 域相同，尽管源字符串不同。

- **正确做法：** 从规范化后的值集合推导「无限制/有限制」标志。
- **发现位置：** `apps/cli/src/schedule.ts` —— DOM/DOW 的 OR 语义让 `*/1` 使一个
  指定星期几的自治任务每天都运行。

## 32. 可变的序号不是持久身份

索引会在插入/删除后变化。按显示索引排队两个变更操作，重新编号后第二个操作
可能作用到另一个条目上。

- **正确做法：** 使用稳定 id/内容指纹，串行化变更，身份存疑时失败关闭
  （fail closed）。
- **发现位置：** `apps/desktop/src/views/MemoryView.tsx` —— 并发删除记忆事实。

## 33. 不可信数据中不能包含它自己的围栏分隔符

把某个提示词区段标注为「不可信」毫无意义——只要负载文本能闭合该区段，
就能在区段之外接续指令式文本。

- **正确做法：** 对每一个被插值的键和值，按分隔符语法进行编码。
- **发现位置：** `apps/server/src/triggers.ts` —— webhook 标题可以向 headless
  编辑任务注入 `</untrusted-event-data>`。

## 34. 在每次可观测的用量更新时聚合成本

只在完成时记账，会在后台子任务被中止或比父任务活得更久时丢失已经计费的工作。

- **正确做法：** 立即合并单调递增的累计增量，完成时绝不把同一个子任务的
  总量再加一遍。
- **发现位置：** `packages/core/src/agent/loop.ts` —— 后台子代理用量。

## 35. 同步调用无法执行外部的墙钟预算

在一个可能灾难性回溯的正则或阻塞式子进程等待外面检查 `Date.now()`，
无法在调用占有线程期间打断它。

- **正确做法：** 执行前先拒绝不安全的正则语法；子进程方法要运行在自有的
  进程组中，配备有界的输出排空、取消轮询和内部截止时间。
- **发现位置：** `apps/server/src/files.ts` 的正则搜索和 Rust runtime 的 Git 调用。
- **同类问题：** Core 的 `search_text` 在 Node 主线程上执行任意正则表达式；
  应在构造表达式之前拒绝反向引用和有歧义的嵌套量词。
- **同类问题：** 自定义命令的超时只杀掉了 shell，并把已捕获的 stdout 当作成功。
  应拥有整个进程组、限制输出、终止后代进程，并拒绝任何超时或非零退出。

## 36. 持久化缓存命中需要完整的契约校验

时间戳有效不代表缓存的负载有效。不完整的对象、非有限或不可能的计数器都可能
让消费者崩溃或污染预算。

- **正确做法：** 校验完整的响应形态和数字不变量；任何不匹配都当作缓存未命中。
- **发现位置：** `packages/core/src/provider/cache.ts` —— 缓存的 `ChatResponse` 数据。

## 37. 传输成功不等于验证成功

一个合法的工具结果信封，描述的可能是一个失败的或仍在运行的进程。

- **正确做法：** 在记录验证或 lint 成功之前，要求满足操作特定的成功条件，
  包括前台完成和退出码为零。
- **发现位置：** `packages/core/src/agent/loop.ts` —— 非零退出和后台命令曾经
  满足了 verify/lint 完成门槛。
- **同类问题：** 数字选项解析器必须对非法输入抛出异常，而不是返回
  `undefined`，Commander 会把后者当作选项未提供。

## 38. 环境变量中的命令字符串需要 argv 解析器

`EDITOR="code --wait"` 是「命令加参数」，不是一个可执行文件名。按空白分割
还会破坏带引号的路径和参数。

- **正确做法：** 在不调用 shell 的前提下把引号与转义解析成 argv；拒绝畸形的
  引号写法，并用解析出的参数启动可执行文件。
- **发现位置：** `apps/tui/src/app.tsx` —— `/memory edit` 和 `/config edit` 把
  完整的 `$EDITOR` 值当作可执行文件传了进去。

## 39. 内部错误必须在公共边界处完成转换

重构一个门面（facade）可以保持返回值不变，却意外暴露一种新的异常类型。
把领域错误映射为 HTTP 状态码或 CLI 输出的调用方，随即把一个已处理的
客户端错误变成了未知的服务器故障。

- **正确做法：** 在拥有该服务的边界处捕获底层的策略、文件系统和传输错误，
  转换为既定的领域错误；用契约测试覆盖状态码、错误码和消息语义。
- **发现位置：** `apps/server/src/file-upload-raw.ts` —— 共享路径守卫的
  `ToolError` 从 `saveUpload` 中逃逸，而不是抛出文档规定的 `UploadError(400)`。

## 40. 便利型解码器可能接受畸形编码

许多标准库解码器有意设计得宽容。解码成功并不能证明不可信输入遵守了
声明的线上格式（wire format）。

- **正确做法：** 校验语法，并与规范化的往返（round trip）结果比对，
  再接受编码输入。
- **发现位置：** `apps/server/src/file-upload-raw.ts` —— `Buffer.from(..., "base64")`
  忽略非法字符，接受了畸形的图片上传。

## 41. Effect 清理在依赖变化时也会运行，而不仅是卸载时

把进程级的清理放在一个依赖可变状态的 effect 里，意味着每次重新运行之前
都会先执行这段清理。而兄弟 effect 未必会重新运行来恢复被清掉的东西。

- **正确做法：** 依赖变化触发的清理只作用于该 effect 自己的资源，组件级的
  清理放在单独的卸载 effect 中。
- **发现位置：** `apps/tui/src/use-terminal-lifecycle.ts` —— 切换鼠标捕获会清掉
  终端标题，直到另一个标题依赖发生变化才恢复。

## 42. 渲染过程中不得修改已有的异步协调器

React 的并发渲染可能被放弃。在渲染期间修改一个长生命周期的协调器，
会让一次未提交的渲染使屏幕上仍在显示的状态失效。

- **正确做法：** 用 `useMemo` 创建与身份绑定的协调器，通过 ref 更新回调而
  不改变所有权，并在 effect 清理中使旧实例失效。
- **发现位置：** `apps/desktop/src/views/use-workspace-async.ts` —— 工作区切换
  在渲染提交之前就修改了上一个协调器。
- **同类问题：** 工作区打开、记忆变更/统计和余额请求在其所属工作区已切换后
  才提交；每一次完成和回滚都必须绑定到 generation 或工作区身份。
- **同类问题：** 安全、设置/MCP、代理、会话、诊断、演进、技能、文件、diff、
  待办以及嵌套的记忆控件，都允许迟到的成功、错误、清理或定时器回调去修改
  新激活工作区的 UI。要给每一次完成加作用域，而不只是请求 URL，还要包括
  报告下载这类浏览器侧副作用。

## 43. 回退分支必须保留请求的全部谓词

慢路径或绕过缓存的路径实现的仍是同一个请求。直接返回其原始中间结果会
悄悄丢掉过滤、排序或分页。

- **正确做法：** 让缓存分支和非缓存分支共享后处理逻辑，或显式重复每一个
  谓词并配以边界测试。
- **发现位置：** `apps/server/src/file-scan-search.ts` —— 一次扩大范围的
  非缓存遍历忽略了调用方的 `q` 过滤条件。

## 44. 先认证，再暴露资源状态

在认证之前就查找资源或检查其是否启用，会让状态码变成一个枚举探测器
（enumeration oracle），即便受保护的操作本身从未执行。

- **正确做法：** 先对着一个不泄露信息的兜底对象校验凭据，并对缺失、禁用、
  启用三种资源返回同样的认证失败。
- **发现位置：** `apps/server/src/routes/triggers.ts` —— 伪造的 GitHub webhook
  请求头在 HMAC 校验之前就能区分未知、禁用和启用的触发器 id。

## 45. 备选执行后端必须保持安全策略

更快的后端如果其协议无法表达当前生效的沙箱、权限、取消或资源策略，
就不能互换使用。

- **正确做法：** 路由到具备策略能力的后端，或者直接拒绝；绝不悄悄丢弃
  不受支持的约束。
- **发现位置：** `packages/core/src/tools/builtins/command.ts` —— Rust Runtime
  路径绕过了处于激活状态的操作系统命令沙箱。

## 46. 不透明分页需要进度守卫

忽略 `nextCursor` 会悄悄隐藏数据；无限信任它则会让一个畸形的服务器制造
死循环。

- **正确做法：** 消费每一个不透明游标，包括合法的空字符串；只有游标缺席才
  意味着分页结束。拒绝重复的游标，并施加一个有文档说明的页数/条目上限。
- **发现位置：** `packages/core/src/mcp/client.ts` —— 工具、资源和 prompt 的
  发现只返回了第一页。

## 47. 声明的能力是必须兑现的承诺

声明一项协议能力（capability）会导致对端发送请求并等待答复。声明一个
只实现了一半的特性比不声明更糟。

- **正确做法：** 只声明当前传输确实能服务的能力；保留协商出的版本/能力状态
  供后续请求使用。
- **发现位置：** `packages/core/src/mcp/http.ts` —— HTTP 声明了 roots 能力，
  却可能丢弃一个请求作用域的 `roots/list`，让一个遵循协议的服务器死锁。

## 48. 记忆化的启动 Promise 必须在每条失败路径上重置

`promise.then(success, failure)` 的双回调形式不会把 `success` 中抛出的异常
送进那个 `failure` 回调。即便传输层的本意是让下一次调用重试初始化，
被 reject 的 Promise 也可能永远留在缓存里。

- **正确做法：** 在握手校验之后追加一个最终的 `.catch(...)`，并清除所有
  部分生命周期状态（会话 id、协商版本、缓存的 Promise）。
- **发现位置：** `packages/core/src/mcp/http.ts` —— 一次畸形的 initialize 结果
  永久毒化了客户端，并保留了服务器的部分会话 id。

## 49. 宽泛的可写父目录会覆盖受保护的嵌套根目录

允许临时目录保持可写，也就允许了其下的每一个工作区可写——除非嵌套的
工作区被再次显式保护。

- **正确做法：** 调整挂载/profile 规则的顺序，让更窄的工作区策略胜过宽泛的
  临时目录许可，并直接测试「工作区位于临时目录内」这一情形。
- **发现位置：** `packages/core/src/tools/os-sandbox.ts` —— 位于 `/tmp` 或
  `TMPDIR` 之下的 `read-only` 工作区继承了父目录的写权限。

## 50. 除非 UI 做了排队，交互式提示就是一个串行化资源

并发启动多个各自等待确认的操作，可能覆盖唯一的「待确认提示」槽位，
让被顶掉的 Promise 永远得不到 resolve。

- **正确做法：** 串行化交互式授权，然后按请求的并发度运行已批准的工作；
  使用完成驱动的调度器，避免一个慢的兄弟任务用批处理屏障卡住无关的
  就绪分支。
- **发现位置：** `packages/core/src/agent/loop.ts` —— 并发的编辑团队成员竞争
  单槽位的权限 UI，且 `Promise.all` 卡住了新就绪的分支。

## 51. 异步结果需要完整的可变目标身份

当结果要写入某个标签页时，只检查工作区 id 是不够的：用户可以在同一工作区内
切换标签页，也可以在完成之前 A→B→A 地切换。

- **正确做法：** 同时捕获工作区和标签页身份，支持取消，省略为空的可选参数，
  并在请求失败时保留重试状态。
- **发现位置：** `apps/desktop/src/views/SettingsView.tsx` —— 一个缓慢的 MCP
  Prompt 可能覆盖另一个标签页的草稿，并把空的可选参数当作有效值。

## 52. 外部上下文在模型边界处必须始终以「数据」的形态可见

把第三方文本直接拼到用户任务后面，会让其中嵌入的指令获得与用户请求相同的
视觉权威，即便工具权限仍然受控。

- **正确做法：** 把外部内容序列化进一个显式的不可信数据信封中，提示词中
  省略不可信的传输错误信息，并强化系统规则。
- **发现位置：** `apps/server/src/agent.ts` 和 `apps/tui/src/app.tsx` —— MCP
  Resource 内容被直接拼接到任务消息上。
- **还发现于：** `packages/core/src/agent/auto-loop.ts` —— 验证器输出在没有显式
  不可信数据信封的情况下被拼进 Loop 后续提示词，使仓库可控的诊断文本看起来像
  用户指令。

## 53. 启动阶段的取消与清理先于主操作开始

只挂在最终运行上的取消，无法阻止组装该运行期间发生的 provider/工具发现。
后续的构建失败还可能泄漏此前已获取的资源。

- **正确做法：** 把运行 signal 贯穿到发现阶段，在每一次抛出时销毁部分构建的
  客户端，并在构建客户端之前隔离畸形条目。
- **发现位置：** `packages/core/src/mcp/tools.ts` 和 `apps/server/src/agent.ts` ——
  MCP 发现忽略了取消，且部分完成的 Agent 组装可能泄漏客户端。

## 54. 取消的可观测性从请求派发之后才开始

初始化期间的 abort 可以正确地阻止真正的请求被发出。此时使用固定定时器的
测试可能会去期待一个服务器从未见过的请求 id 的取消通知。

- **正确做法：** 断言传输层取消的副作用时，先同步等到服务器收到目标请求
  再 abort；延迟应从 abort 时刻起算，而不是从客户端构建或握手启动起算。
- **发现位置：** `packages/core/tests/mcp/http.test.ts` —— 一个 25 ms 的定时器
  与 initialize 握手竞速，使通知断言变得不确定。

## 55. 预算要覆盖完整的线上请求，而不仅是显眼的负载

消息并不是模型请求的全部：工具定义及其 JSON schema 每一轮都会被序列化，
可能占据上下文窗口的大头。

- **正确做法：** 在 provider 边界同时估算消息和已声明的工具；为两者都预留
  空间，并对过大的工具目录做确定性的收窄。
- **发现位置：** `packages/core/src/agent/loop.ts` —— 上下文压缩（compaction）
  只考虑了消息，因此一个巨大的 MCP 目录可能在 UI 报告空间充裕之后
  仍然超出窗口。

## 56. 每一次模型调用都应纳入用量与预算核算

辅助性的摘要、抽取、排序或评审调用同样消耗 token 和金钱，即便它们不是
主 agent 回合。

- **正确做法：** 从每一个成功的 provider 响应中返回用量，包括语义上畸形的
  响应，并在最终报告之前完成聚合。
- **发现位置：** `packages/core/src/agent/context.ts` 与 `memory/extract.ts` ——
  LLM 压缩和记忆抽取丢弃了用量，低估了 Loop 成本。

## 57. 出现在提示词中不等于被使用

把一条记录放进提示词，并不能证明它影响了模型的工作。把「曝光」当作「使用」
会让留存和质量指标自我实现。

- **正确做法：** 把被动曝光、显式检索和确证使用记录为三个独立的计数器；
  按真正想要的信号来做裁剪和评估。
- **发现位置：** `packages/core/src/agent/loop.ts` —— 每一条注入的记忆事实都被
  标记为已使用，即便它与任务毫不相关。

## 58. 追加式日志只能恢复其最长有效前缀

跳过一条畸形 JSONL 记录并接受后续行，会让「撕裂或损坏写入之后的状态」
覆盖最后的持久化状态。合法的 JSON 标量和不符合 schema 的对象同样是损坏，
不是可以无害忽略的记录。

- **正确做法：** 按顺序解析并校验每一条记录，在第一个 JSON 或 schema 失败处
  停止，并在接受下一个事件之前强制单调的序号/时间戳不变量。在后续追加之前，
  先原子地截断无效后缀；否则之后所有的有效记录都将永远不可达。
- **发现位置：** `apps/server/src/run-ledger.ts` —— 运行快照和 WS 重放事件原本
  会跳过中间的畸形记录，继续读取后面的状态。

## 59. 终态生命周期必须拒绝迟到的异步状态迁移

取消可能与 provider、子进程或游离的完成事件竞速。迟到的回调仍会执行，
但它已经无权决定生命周期。

- **正确做法：** 在中央 store 中强制执行允许的状态迁移；一旦运行已 succeeded、
  failed 或 cancelled，忽略与之冲突的终态更新。
- **发现位置：** `apps/server/src/run-ledger.ts` —— 一个后台运行可能先被标记为
  cancelled，随后又被一个迟到的完成事件改写为 succeeded。

## 60. 自主变更入口必须要求有限的显式预算

后台或 UI 触发的编辑路径即使最终调用同一个 Agent 实现，也可能绕过 CLI 的
成本护栏。

- **正确做法：** 在每一个自主变更边界校验有限的正预算，监听累计用量事件，
  并把取消传播进活跃的 provider/工具图。
- **发现位置：** Desktop 安全中心的自动修复最初接受验证命令却没有
  `maxCostUsd`，使编辑 Agent 不受任何限制。

## 61. 感知失败的 UI 完成回调不得执行成功清理

一个捕获错误后正常 resolve 的异步辅助函数，会让下游的 `.then(...)` 处理器
无法区分成功与失败。

- **正确做法：** 返回显式的成功结果（或重新抛出），并且只在确认成功之后
  才关闭对话框或清除用户输入。
- **发现位置：** 失败的 Finding 生命周期/修复请求关闭了对应的 Desktop 对话框，
  丢弃了用户的输入，错误却只显示在模态框背后。

## 62. 聚合结论必须从已校验明细推导，不能信任模型自报

结构化模型响应可能声称 `complete: true`，但自身标准记录仍是未满足、缺失、
重复或未知。

- **正确做法：** 校验标识符精确覆盖，并从必需明细记录推导聚合结果。
  聚合值不一致时拒绝；结构化证据无法解析时按关闭失败处理。
- **发现位置：** `packages/core/src/agent/loop-requirements.ts` —— 验收完成状态必须
  从冻结的必需标准计算，不能信任模型提供的布尔值。

## 63. 中间输出不等于阶段已经成功完成

模型可能先输出看似合法的文本，随后在工具调用或下一次 provider 回合失败。
如果不检查终态就保留前面的文本，会把未完成阶段误判成成功。

- **正确做法：** 只有同一次运行发出成功终态事件后，才接受分析/评审输出；
  取消、预算中止和 `session.failed` 都必须让该阶段保持未完成。
- **发现位置：** `packages/core/src/agent/auto-loop.ts` —— 失败的需求验收仍可能从
  较早的 `model.message` 判定通过；取消的分析也可能持久化 fallback，导致恢复后
  永远不再真正分析。

## 64. 批准只能作用于调用方已经观察到的产物

产物尚不存在时发送的批准参数，不能证明调用方看过最终要批准的准确内容。

- **正确做法：** 把批准绑定到从持久化状态加载的标识/版本；新生成的产物必须
  先返回，再由后续调用批准。
- **发现位置：** `packages/core/src/agent/auto-loop.ts` —— confirm 模式的需求可以
  在同一次调用中生成并批准。

## 65. 进度指纹必须包含干净的已提交变更

一次运行如果修改并提交了文件，运行前后的工作树状态都可能是干净的。只看脏路径
会把真实进展误判成没有变化。

- **正确做法：** 除脏文件和未跟踪内容外，还要纳入仓库 `HEAD`/tree 身份；
  非 Git 工作区保留内容指纹 fallback。
- **发现位置：** `packages/core/src/agent/auto-loop.ts` —— 已提交的修复在验证诊断
  不变时可能错误触发 `no_progress`。

## 66. 有界事件流不是持久 UI 状态

淘汰旧进度事件不能顺带擦除后续操作所需的当前规格、结果或会话身份。

- **正确做法：** 把持久工作流字段独立于有上限的展示事件流进行归约，并从终态
  快照重新补齐。
- **发现位置：** `apps/desktop/src/lib/loop.ts` 与 `tabs.ts` —— 大量 Loop 输出可能
  淘汰需求，并让后续聊天脱离 Loop 会话。

## 67. 数值等价不代表输入语法等价

`Number()` 会接受十六进制等语法，而 CLI 文档约定的十进制解析器会拒绝，造成
不同界面行为漂移。

- **正确做法：** 先校验完整文本语法，再做数值转换和有限值/范围校验。
- **发现位置：** TUI 与 Desktop 的 Loop 控件会接受 `0x10` 迭代数，而 CLI 拒绝。

## 68. 陈旧锁恢复本身也是协调操作

比较锁内容与删除路径是两个独立的文件系统动作。两个恢复者可能都确认旧 owner
已经失效，随后其中一个却删除另一个刚创建的新锁。

- **正确做法：** 用排他的恢复标记串行化陈旧锁恢复；持有恢复权后重新读取，
  并让新候选在恢复期间主动让出。
- **发现位置：** `packages/core/src/agent/loop-state.ts` —— 并发恢复死亡的 Loop
  lease 时可能同时放进两个 owner。

## 69. 路径权限规则匹配前必须先规范化别名

`src/../secrets/key` 之类的词法别名可以跨越允许/拒绝边界，同时保留误导性的前缀。

- **正确做法：** 在精确或子路径匹配前，同时规范化分类得到的路径和配置规则路径。
- **发现位置：** `packages/core/src/tools/permissions.ts` —— 允许的 `src` 前缀可能
  放行 `src` 外的别名路径，或掩盖被拒绝的目录。

## 70. 必须在文件系统变更边界重新校验目标

路径校验与文件变更是两个操作。目标可以在校验后、按路径写入前被替换为外部符号链接。

- **正确做法：** 使用禁止跟随符号链接的方式打开，在最终检查点前后比较父目录和
  文件身份，并通过已校验的文件描述符执行变更。
- **发现位置：** `write_file` 和 `apply_patch` 可能跟随工作区校验后被换入的叶节点
  符号链接。

## 71. 内部持久化必须与工具写入采用相同的物理边界

当中间状态目录是符号链接时，工作区内的状态路径并不一定真的位于工作区内。

- **正确做法：** 所有持久化写入都通过物理写目标守卫解析，包括追加和原子重写。
- **发现位置：** memory 和 evolution store 可能通过符号链接的
  `.seekforge/memory` 或 `.seekforge/evolution` 目录写到工作区外。

## 72. 工具审批不能授权更早发生的插件启动

插件发现本身就可能启动进程或访问远程端点。之后围绕工具调用出现的权限提示，
不能反向授权此前的启动。

- **正确做法：** 自动发现前先要求连接信任；显式管理探测应独立，并由用户主动发起。
- **发现位置：** 未信任的 MCP stdio 服务器会在组装 Agent 时、任何工具级确认出现前启动。

## 73. 同一个终止原因必须映射到同一个协议状态

持久状态记录为取消、实时通道却发送通用失败，会让在线客户端和重连客户端看到不同结果。

- **正确做法：** 从同一个终止原因推导账本状态和发出的错误码。
- **发现位置：** 已中止的 WS Agent 抛出异常时，账本记为 `cancelled`，但在线客户端
  收到 `agent_error`。

## 74. 包装器选项不能隐藏需要分类的子命令

只检查第一个位置参数的命令分类器，会漏掉全局选项之后的危险子命令，尤其是值与选项
分开的写法。

- **正确做法：** 匹配有效子命令前，完整消费包装器的全局选项语法。
- **发现位置：** `git -C . push` 和 `git -c core.pager=cat push --force` 能绕过
  push/force-push 分类。

## 75. 进程内计数器不能生成持久标识符

时间戳加内存计数在多个进程于同一时钟刻度写入同一持久存储时可能冲突。

- **正确做法：** 为持久记录使用跨进程、抗冲突的标识符。
- **发现位置：** 多个 CLI 进程手动添加的 memory fact 可能获得相同 id。

## 76. 独立变更入口必须加入同一个仓库协调器

每连接 busy 标记只能防止该连接内部重叠；WS Agent、后台 Loop、webhook 与安全修复
仍可能同时指向同一个 checkout。

- **正确做法：** 所有 Server 管理的变更操作都通过同一个物理仓库协调器调度，
  并在排队前注册取消控制器。
- **发现位置：** 不同 WS 连接和后台 REST 运行可能与安全/Git 操作并发修改同一工作区。

## 77. 持久化去重仍需要原子 claim

read-check-write JSON 文件可以跨重启保留，但两个进程都可能在任一原子 rename 可见前
读到键不存在。

- **正确做法：** 使用可恢复陈旧状态的跨进程 lease 保护整个 claim/rollback 事务。
- **发现位置：** 两个 Server 实例可能同时接受同一个 GitHub delivery id。

## 78. 可重连客户端必须保留服务器回放游标

仅自动重连 socket 无法找回运行接受后丢失的帧；客户端还需要持久运行身份和最后应用序号。

- **正确做法：** 在工作流状态中保存 `runId + seq`，重连时使用 `afterSeq` 订阅，
  并忽略陈旧或重复运行帧。
- **发现位置：** Desktop 断连时可能丢失 Agent/Loop 终态，导致结果或会话身份不完整。

## 79. 分离任务仍然拥有结果目的地

把执行从前台控制中分离，并不会让其终态输出脱离接收它的标签页或文档。

- **正确做法：** 在分离任务完成前保持目的地存在，或在允许关闭前显式迁移所有权。
- **发现位置：** TUI 在分离 Loop 后允许关闭原标签页，最终结果会被静默丢弃。

## 80. 完整缓冲后的大小检查不构成内存上限

先读取 `arrayBuffer()` 或 `text()`，再检查长度，会让不受信任的对端在守卫执行前
消耗任意内存。

- **正确做法：** 增量消费流，在保留每个 chunk 前统计字节，溢出时取消读取，
  并让超时一直覆盖到响应体读取完成。
- **发现位置：** `web_fetch`、`web_search` 和 MCP HTTP 的普通 JSON/OAuth 响应
  要么在完整缓冲后才检查上限，要么完全没有上限。

## 81. 分隔协议必须在寻找分隔符期间保持有界

只有找到分隔符后才执行的行/帧上限，无法阻止未终止输入无限扩张解析缓冲区。

- **正确做法：** 在读取过程中限制累积；溢出后使用固定大小块丢弃到分隔符，
  使后续帧仍可继续处理。
- **发现位置：** Rust runtime 对 JSONL 请求使用无界 `read_until('\n')`，
  单个超大行即可耗尽子进程内存。

## 82. 传输层终止原因是载荷有效性的一部分

当 provider 表明生成因输出 token 上限而停止时，仅有文本内容不能证明响应完整。

- **正确做法：** 将长度受限响应视为未完成，请求一份可独立使用的完整答案；若已无剩余
  轮次，则明确失败。
- **发现位置：** Agent 曾把 `finishReason: "length"` 响应当作最终报告，静默持久化
  被截断的输出。

## 83. 取消必须覆盖人工交互等待

如果面向用户的 Promise 可能无限等待，只在进入审批或提问前检查 abort signal 并不充分。

- **正确做法：** 让每个权限确认和提问等待都与运行的 abort signal 竞争，并在任一侧
  完成时移除监听器。
- **发现位置：** Agent 等待权限或 `ask_user` 时取消，会让运行及其工作区 lease 卡住。

## 84. 缺失与显式为空的安全策略是不同状态

当字段缺失表示“使用无限制默认值”时，把空白名单归一化为缺失会扩大权限。

- **正确做法：** 在解析、合并和序列化过程中保留显式空集合；只有字段真正缺失时才
  应用默认值。
- **发现位置：** subagent 的 `tools: []` 白名单会变成 `undefined`，从无工具权限扩大为
  全部工具权限。

## 85. 关闭迭代器属于终止生命周期路径

async generator 可能被消费者关闭，而不抛错，也不进入生产者的正常完成分支。

- **正确做法：** 在 `finally` 中落定持久状态并释放资源，同时保留正常终止路径已写入的状态。
- **发现位置：** 提前关闭 Agent 事件迭代器后，执行虽已停止，会话仍持久化为 `running`。

## 86. 结构化数据必须先脱敏再序列化

文本脱敏器作用于已序列化记录时，可能插入换行，或吞掉 JSON 引号等语法。

- **正确做法：** 先递归脱敏字符串叶节点，再对结果结构执行且只执行一次序列化。
- **发现位置：** run-event 在 `JSON.stringify` 后脱敏会破坏 JSONL，也无法可靠移除
  多行 PEM 内容。

## 87. 敏感路径策略必须覆盖所有模型读取入口

只保护文件工具，会让其它上下文构建器和自动放行的搜索命令读取同一个秘密。

- **正确做法：** 对文件工具、`@path` 展开、工作区目录展开和命令自动审批统一应用
  basename 与相对路径策略。
- **发现位置：** `.seekforge/config.json` 和 `triggers.json` 可通过任务引用或显式指定的
  `rg` 命令进入模型。

## 88. 凭据名称匹配需要语义边界

`TOKEN` 这类无边界子串既会漏掉其它凭据名称，也会删除仅仅包含该单词的普通构建设置。

- **正确做法：** 环境清理和输出脱敏共享凭据类别，按分隔符/驼峰边界匹配，并测试非秘密名称。
- **发现位置：** `GITHUB_PAT` 会泄漏，而 `MAX_TOKENS`、`TOKENIZERS_PARALLELISM`
  会被静默删除。

## 89. 删除必须校验每一个物理父目录

当 `subdir` 能成为指向外部目录的符号链接时，安全文件名并不能保证
`root/subdir/file` 安全。

- **正确做法：** 删除与读写采用同一个物理项目路径守卫，并拒绝符号链接父目录和叶节点。
- **发现位置：** run-ledger 压缩可通过 `.seekforge/run-events` 删除外部
  `run-*.jsonl`。

## 90. 进程级 teardown 注册必须可注销

如果正常销毁不移除监听器，每实例 signal listener 会活得比短生命周期客户端更久。

- **正确做法：** teardown 注册返回幂等 disposer，并在所属客户端永久销毁时调用。
- **发现位置：** 每个 RuntimeClient 关闭后仍遗留四个进程监听器及其闭包。

## 91. 选项解析器必须消费选项的完整参数个数

跳过 flag 却未跳过其独立值，会把该值提升为子命令，从而向策略分类隐藏真正操作。

- **正确做法：** 编码哪些全局选项需要下一个参数，包括短/长拼写，并用 TS/Rust parity
  fixture 完整覆盖。
- **发现位置：** Rust 把 `git --git-dir PATH` 后的路径当成子命令，使后续破坏性 Git
  操作通过。

## 92. 原子替换仍要求完整写入

原子 rename 只保证哪个文件可见，不能保证一次 `write` 系统调用消费了整个 buffer。

- **正确做法：** 循环写到所有字节完成，按返回字节数推进，并在 writer 无进展时于
  fsync/rename 前失败。
- **发现位置：** `writeFileAtomic` 遇到短写时可能用截断临时文件替换持久状态。

## 93. 复合命令的 flag 只属于一个 invocation

在整条 shell 行中搜索危险 flag，可能把前一命令参数归给后一命令，并导致独立分类器分歧。

- **正确做法：** 识别命令边界，只在所属 invocation 内分类 flag，并用复合 fixture 固定
  跨语言行为。
- **发现位置：** Rust 把 `echo --force && git push` 中的 `--force` 归给 Git push，
  TypeScript 则没有。

## 94. 策略规范化必须符合 Shell 续行语义

单引号外的反斜杠换行会在 Shell 分词前被移除；把它当成空白会让拆成两行的危险命令隐藏起来。

- **正确做法：** 在策略匹配和单词解析前移除 Shell 续行，保留单引号内的原文，并固定
  跨运行时一致性。
- **发现位置：** `r\\<换行>m -rf` 和 `git pu\\<换行>sh --force` 实际执行为被拒命令，
  却绕过了 TypeScript 与 Rust 分类器。

## 95. 父级取消必须覆盖每一个网络阶段

操作自身的超时不能让工具响应所属 Agent 运行的取消，尤其是在 DNS 尚未解析完成时。

- **正确做法：** 把父级信号连接到请求控制器，并让 DNS、请求和响应体消费都可被中止；
  操作结束时移除监听器。
- **发现位置：** 已取消的 `web_fetch`、`web_search`、`image_analyze` 和 Browser 操作
  仍会继续运行，直到各自独立的超时；Vision 还会在收到响应头后清除超时，使响应体解析
  失去时间上限。

## 96. 仓库配置不得触发启动时执行

配置优先级不代表每一层具有相同信任级别；在用户批准任何动作前，刚检出的仓库仍不可信。

- **正确做法：** 启动时 shell 命令只能来自用户拥有的配置，并使用最小化白名单环境启动。
- **发现位置：** 项目 `.seekforge/config.json` 可在 TUI 打开时立即运行 `statusLine`，
  并从 `process.env` 继承 provider API 密钥。

## 97. 分隔流必须在分隔符前限制 frame

只在找到换行后检查大小的解析器，即使之后会校验每条记录，也会无界缓存未终止记录。

- **正确做法：** 等待分隔符前先限制待处理缓冲区，同时拒绝已终止的超大 frame，
  并立即失败或终止生产者。
- **发现位置：** CLI stream-json 输入和 Runtime stdout JSONL 都可被一条永不结束的行
  耗尽宿主内存；普通 CLI 管道文本也会在没有总量上限的情况下累积全部 chunk。

## 98. 异步 UI 结果必须持续绑定请求身份

props 变化不会取消早期 render 启动的 Promise；迟到结果可能覆盖已属于另一资源的状态。

- **正确做法：** 捕获 generation 或资源身份，在 effect cleanup 中使其失效，并守卫成功、
  错误和 finally 回调。同一资源内还要捕获编辑修订号，防止保存响应隐藏更新的输入。
- **发现位置：** 迟到的文件 A 读取可进入文件 B 编辑器，后续保存会把 A 内容写到 B；
  文件索引加载也存在同类工作区切换竞态；保存完成也可能在用户输入更新、尚未保存的修订后
  关闭编辑器。

## 99. 在通用 JSON 解析之前执行响应大小限制

`Response.json()` 会缓冲完整响应体，无法在字节到达时执行应用层上限。

- **正确做法：** 不受信任的 API 响应先通过共享的有界增量读取器，再解析已受限的缓冲区。
- **发现位置：** Vision API 响应绕过 Web 响应上限，可在 JSON 解析完成前无界消耗内存。

## 100. 快照替换与追加必须共享同一个跨进程租约

原子 rename 只能避免文件撕裂，不能阻止压缩器在另一进程向旧文件追加后用快照覆盖它；
对端追加或替换文件后，本进程缓存的行数也会失效。

- **正确做法：** 让所有追加与压缩共享同一个跨进程租约；在租约内比较低成本文件身份，
  发现对端写入后重新计数。
- **发现位置：** Server run-ledger 并发压缩可能永久丢弃另一进程的运行记录。

## 101. 后台轮询必须负责异常与终态清理

定时器回调抛出的异常会逃逸请求 Promise；资源结束后仍不注销的轮询会持续保留连接状态。

- **正确做法：** 捕获每次轮询，在失败、socket 关闭和观察到终态帧时注销，且只暴露通用传输错误。
- **发现位置：** WS 运行订阅可能因回放读取异常而使进程崩溃，或在交付终态后永久轮询。

## 102. 轮询游标必须让空闲与增量工作保持低成本

每次定时器触发都从零开始扫描持续增长的追加日志，空闲时仍是 O(N)，长运行累计为 O(N^2)，
即使响应分页有上限也无法避免。

- **正确做法：** 同进程追加使用直接通知；低频跨进程兜底先做 O(1) 文件身份比较，
  仅在发生变化后读取有界页面。
- **发现位置：** 实时 WS 订阅每 25 ms 重新解析完整 run-event JSONL，没有新事件时也一样。

## 102. 取消必须持续拥有所有子操作，直到清理完成

只在 turn 开始时检查 abort，无法覆盖 hook、响应后的处理，也无法覆盖 async iterator
关闭后仍在运行的工具。

- **正确做法：** 将调用方取消桥接到 run 自有 signal，并传给所有子操作；在外部可观察的
  usage 事件后重新检查；对可能忽略信号的 provider Promise 主动竞速；释放 session lease
  前等待活动工具和 subagent 完成清理。即使进入“信号已取消”分支，也必须观察生产者
  Promise，避免同步拒绝逃逸为全局未处理错误。
- **发现位置：** 预算取消后仍可能提取 memory 并完成 session；`iterator.return()` 也可能
  在工具退出前释放 lease；工具级 pre/post hook 也会忽略运行信号；后台 dispatch 还会
  错误地跟踪短暂的启动 Promise，而不是底层子运行。

## 103. Agent 生成态不属于工作区进展

如果进展指纹包含 Agent 自己的日志、memory candidate 或 session 状态，即使产品源码和
verifier 结果完全没变，指纹也会变化。

- **正确做法：** Git 路径和 fallback 路径都要排除所有生成态目录。
- **发现位置：** memory extraction 每轮修改 `.seekforge/memory/candidates.jsonl`，导致
  Loop 的 `no_progress` 守卫无法触发。

## 104. 并发写入者需要工作区隔离或串行化

两个 Agent 可以基于同一个旧快照成功写入，后写结果会静默覆盖先写结果。

- **正确做法：** 使用隔离 worktree 或内容版本 CAS；若项目尚无这类机制，应保守地
  串行执行共享工作区的 edit-mode Agent，同时保留只读 Agent 的并发能力。
- **发现位置：** 独立的 `dispatch_team` edit 成员可能互相覆盖修改，却都报告成功。

## 105. 配置型子进程只能获得最小权限环境

命令由用户配置，并不表示父进程中的所有 provider token 和主机凭据都与该命令相关。

- **正确做法：** 通过共享 secret scrubber 构造子进程环境，再只添加子协议明确需要的
  元数据变量。
- **发现位置：** hook 继承完整 `process.env`，在任何工具权限边界之前就能读取 provider
  API key。

## 106. 授权必须固定到物理资源身份

批准一个逻辑路径，并不等于批准该路径上的符号链接之后可以指向的任意目标。

- **正确做法：** 授权目录时只规范化一次，并保存该物理路径供后续读取；不要再次解析
  原来的逻辑别名。
- **发现位置：** `/add-dir` 的符号链接在批准后可以被重新绑定，使 `@` 引用读取另一个
  外部目录。

## 107. 辅助工作区状态同样需要路径约束与协调

便捷小文件仍是写入面：普通的读取-修改-写回会跟随仓库符号链接、暴露外部内容，或与
正在运行的 Agent 竞争。

- **正确做法：** 拒绝符号链接状态父目录和叶文件，通过 no-follow 文件描述符读取，
  原子替换完整文件，并让每个 UI/API 写入口获取共享 workspace/repository guard。
- **发现位置：** TUI 和 Server 的 todo 操作可能读取或覆盖外部
  `.seekforge/todos.md`，并且写入时没有 Agent 工作区 guard；TUI 还会把失败的写入
  报告成成功修改。

## 108. 配置型子进程需要有界的进程树所有权

只给直接 shell 设置超时并不完整：后代仍可持有管道；`error` 事件也不等于成功 EOF。

- **正确做法：** 异步运行受控进程组，限制捕获字节，失败时销毁管道，先终止再升级杀死
  整个进程组，并且只在确认整个进程组都已退出后取消延迟强杀定时器。
- **发现位置：** REPL shell 展开和 TUI 状态栏会因后代管道挂起或无界收集输出；stdin
  错误会返回部分 prompt，状态栏命令还会阻塞渲染。

## 109. 在破坏客户端状态前执行传输上限

只在服务器限制 frame，会让客户端先清空或追加本地状态，再发现请求根本无法通过传输层。

- **正确做法：** 共享协议常量，按实际序列化 frame 计量，在发送或清空草稿前拒绝，并在
  编码二进制前预检大小。
- **发现位置：** 超大的 Desktop task/Loop frame 会先清空草稿再被 WebSocket 拒绝，
  超大图片也会先做 base64 膨胀。

## 110. 异步 UI 结果必须绑定编辑版本

标签页身份相同，并不能证明延迟返回的转换文本仍属于当前草稿。

- **正确做法：** 异步工作前记录资源级 revision，仅在身份和 revision 都未变化时应用结果。
- **发现位置：** 延迟的自定义命令展开会覆盖同一 Desktop 标签页中稍后输入的文本。

## 111. 子进程从 spawn 成功的一刻起就必须被持有

进程在 readiness 成功前已经存活；若只在启动完成后登记，Exit/清理会有一段时间找不到它。

- **正确做法：** spawn 后立即在生命周期锁内登记 starting 状态，再把同一个受控子进程
  转为 running，或终止它。
- **发现位置：** Desktop 的 Exit 与 readiness 竞争时可能泄漏 sidecar。

## 112. 终态事件必须兼容迭代器安全关闭

在异步生成器 `finally` 中继续 `yield`，会让 `return()` 以 `done:false` 返回，并把释放流程
无限期挂起。

- **正确做法：** `finally` 只等待清理、不产出；在成功/失败事件前显式排空子任务事件，
  保证 session 终态事件最后出现。
- **发现位置：** 后台 dispatch 的用量/文件可能漏出最终报告或出现在完成事件之后；提前
  停止消费还可能卡住 lease 与 hook 释放。

## 113. 重写后的工具输入才是有效安全主体

只验证、授权原始参数，会使 hook 重写后的路径/命令与执行、审计和验证不一致。

- **正确做法：** 对替换输入统一执行 schema 校验、重新分类、重新授权、执行、报告和审计；
  无效替换直接拒绝。
- **发现位置：** `preToolUse.updatedInput` 仍保留原权限元数据与验证命令，格式错误的替换
  还会静默执行原参数。

## 114. 结构化状态需要校验、串行化和物理写入

JSON 能解析不代表值有限且有界；逻辑路径检查也无法抵抗并发写入或符号链接换绑。

- **正确做法：** 校验每个持久化字段，让计数饱和，跨进程序列化读改写事务，通过验证过的
  no-follow 描述符读取，并经重新验证的父目录原子替换。
- **发现位置：** memory 元数据接受非有限值，并发更新会丢失，memory/candidate/archive/
  summary 写入还保留符号链接 TOCTOU 窗口。

## 115. 流协议需要累计预算和时间预算

逐行上限无法限制持续推进的响应，JSON 数字字段也可能带来不安全整数或非有限运算。

- **正确做法：** 限制原始字节、每个累计字段和项目数量，同时设置 idle 与总时限，失败时
  取消 reader，并在成本计算前验证 usage。
- **发现位置：** provider SSE 可跨合法行无限增长或持续推进而永不结束；非流式/错误
  body 会无字节上限地缓冲，异常 token usage 还会生成不安全成本。
